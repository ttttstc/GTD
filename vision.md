# GTD Master Pipeline — Vision

> 一套文件与 Skill 驱动的元编排框架，让 AI Agent 在动手之前先"想清楚、拆清楚、验得过"，再把具体执行分派给已有的专项工作流。

---

## 我们要解决什么问题

AI 辅助工作流的执行能力已经不差——编码交付、sprint 管理、写作、调研等专项 pipeline 各司其职。但所有这些工具都有一个共同的盲区：**没有人先问"这件事该不该做、该怎么拆、怎么算做完"**。

现状是：领导一句话丢过来，Agent 立刻开始写代码写文案。结果要么做偏了回头返工，要么做着做着范围膨胀停不下来，要么做完了谁也说不清算不算做完。

缺的不是执行力，是**结构化的思考力和闭环验证机制**。

## 我们相信什么

**先想后做，想清楚本身就是交付。** 一份定义清晰的目标文档和一份带验收标准的阶段规划，即使后续全部人工执行，也比直接开干的成功率高一个量级。

**方法论可以被工程化。** 优秀的人做事之前脑子里跑的那套"该问什么、该砍什么、该先做什么"的决策过程，可以被结构化为 Skill，让 AI 复现。

**文件即编排，LLM 即引擎。** 不需要构建 runtime、调度器、状态机。一组设计精良的 Skill 文件 + LLM 的推理能力，就足以驱动从目标澄清到分阶段执行的完整流程。

**能力生态渐进生长。** 框架的价值不依赖子工作流的完备性。执行层的能力可以从零开始，随 Skill 生态的丰富逐步补齐。

## 框架定位

GTD Master Pipeline 是各类专项工作流的**上层**。它不替代任何已有工具，而是在它们之上提供一个"战略规划→任务编排→验收闭环"的元层。

```
GTD Master Pipeline
  ├── 想清楚（目标澄清 + 方法论审视）
  ├── 拆清楚（动态生成带验收门的多阶段 Pipeline）
  └── 做到位（调度子工作流执行 + 质量门检查）
        ├── 软件开发工作流
        ├── 文档写作工作流
        ├── 深度调研工作流
        ├── 营销推广工作流
        └── ...（渐进补齐）
```

## 约束与边界

### 做什么

- 强制用户在动手前想清楚目标、终态、验收标准和边界
- 基于目标定义动态生成多阶段工作流，每个阶段可评估可观测
- 为每个阶段设定验收门（数据指标型或人工确认型），支持 pass/fail/pivot
- 定义子工作流的接口契约，使已有 Skill 可即插即用
- 支持加载优秀方法论 Skill 作为思维审视工具，任何高质量的方法论 Skill 均可集成

### 不做什么

- **不做 Runtime**。不构建调度服务、状态机引擎、消息队列。全部通过文件和 Skill 驱动
- **不做具体执行**。不自己实现编码、写作、设计等能力。这是子工作流的事
- **不做 Agent 协作协议**。不涉及多 Agent 通信、A2A、MCP server。执行者始终是读取 Skill 的单个 LLM 实例
- **不做平台绑定**。不依赖 GitHub、Claude Code、任何特定 IDE。Skill 文件可在任何能读 Markdown 的 Agent 平台运行
- **不重复造轮子**。业界已验证的设计模式（design doc 状态传递、进度锚点文件、结构化评审机制等）直接借鉴

### 关键设计约束

- **框架 Skill 总 token < 8K**。主 Skill + 当前活跃阶段 Skill 的合计 token 不得挤压执行上下文。渐进式加载，用到哪层加载哪层
- **阶段数 3-7 个**。超过 7 个必须审视是否过度拆分，少于 3 个审视是否遗漏关键步骤
- **每个阶段必须有验收门**。无验收标准的阶段不允许进入 Pipeline
- **简单任务不过度流程化**。复杂度判定为"简单"的目标跳过多阶段编排直接执行

## 成功的样子

一个使用了 GTD Master Pipeline 的人会说：

- "以前拿到任务就开干，现在会先花 10 分钟想清楚再动手，反而更快完成。"
- "每个阶段做完心里有数，不用猜'差不多了吧'。"
- "做到一半发现方向不对，pipeline 里有 pivot 机制，不用从头来过。"
- "团队新来的人，读一遍 mission-brief 和 pipeline.md 就知道这件事的全貌。"

## 成功标准

- 一个中等复杂度的真实任务，能在 3 轮对话内完成目标澄清+阶段规划，产出可执行的 Pipeline
- 产出的 Pipeline 至少 2 个阶段可对接已有子工作流自动执行
- 框架文件总 token 量 < 8K，不挤压执行上下文
- 使用者反馈："比直接开干少走了弯路"或"目标更清晰了"
